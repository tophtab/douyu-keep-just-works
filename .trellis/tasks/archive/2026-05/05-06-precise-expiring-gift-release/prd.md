# brainstorm: precise expiring gift release

## Goal

把现有“临期任务”的释放策略从“最早过期时间进入阈值后，按当前总荧光棒库存一次性赠送”升级为“只释放临期窗口内的可赠送礼物数量”，减少浪费，同时避免把尚未临期的库存提前全部送掉。

## Why

当前临期任务 MVP 的触发逻辑足够简单，但行为偏粗：

* 只面向荧光棒 `id=268`。
* 会扫描所有可见荧光棒行，汇总总 `count`。
* 会取所有可见荧光棒行中的最早 `expireTime/met/endTime`。
* 一旦最早过期时间进入阈值，就按总库存计算赠送。

这意味着如果未来背包里出现多个过期批次，例如：

* A 批：20 个，明天过期。
* B 批：100 个，下周过期。

当前逻辑会因为 A 批临期而把 `120` 个都纳入本次赠送计划。新需求希望只处理 A 批这类临期库存。

## What I already know

* 用户更关心“把临期内的礼物赠送掉，不要浪费”，而不是“把所有东西全部赠送掉”。
* 新策略不应只限荧光棒；其他背包礼物如果有明确过期时间且可安全赠送，也应能纳入临期释放范围。
* 荧光棒可能存在多个过期时间；即使当前实测样本只显示单条 `id=268`，实现也应按多行/多批次设计。
* 当前项目 `getGiftStatus()` 已能聚合多条 `id=268` 并取最早绝对过期时间，但没有保留每条背包行的独立数量和过期时间。
* DouyuEx `BagInfo` 按斗鱼背包接口 `data.list[]` 行级别展示 `expiry/count/value/intimacy`，但没有解析 `batchInfo`，也没有证明荧光棒底层一定是统一过期或稳定多批次。
* 斗鱼背包 payload 里已观察到 `expiry=5` 与 `met=1778428799` 同时存在；`expiry` 更像展示用剩余天数，`met` 是可用于精确判断的绝对过期时间。
* 当前项目已记录可能的绝对过期字段：`expireTime`、`expire_time`、`expireAt`、`expiresAt`、`met`、`endTime`。

## Desired Behavior

临期释放任务应按“背包可见行/可见批次”计算本次可释放数量：

1. 获取背包明细，保留每个可见礼物行的 `giftId/name/count/expiry/expireTime/batchInfo` 等信息。
2. 归一化每个礼物行的绝对过期时间。
3. 只筛选进入临期阈值的礼物行。
4. 本次任务只把这些临期行的 `count` 纳入可赠送预算。
5. 尚未进入临期阈值的同行或其他行不纳入本次赠送预算。
6. 对无法确认过期时间的礼物，不自动赠送，只在状态里展示“过期时间不可用”。
7. 对无法确认可安全赠送的礼物，不自动赠送，只展示或记录跳过原因。

## Requirements

### Backpack Parsing

* 新增结构化背包明细解析能力，不只返回 `GiftStatus { count, expireTime }`。
* 每个背包行至少保留：
  * `giftId`
  * `name`
  * `count`
  * `expiry` / `expiryDays`（展示用，若存在）
  * `expireTime`（绝对时间戳，若可从 `met` 等字段归一化）
  * `batchInfoPresent`
  * `isValuable`
  * `price`
  * `intimacy`
* 对同一 `giftId` 的多条记录，状态展示应保留行级明细，同时可以提供聚合摘要。
* 保留现有 `getGiftStatus()` 兼容行为，避免破坏概况页和现有任务；新的临期释放逻辑应使用更细的背包明细 API。

### Expiring Selection

* 新增“临期候选行”筛选逻辑：
  * 有正数 `count`。
  * 有可归一化的绝对 `expireTime`。
  * `expireTime - now <= thresholdHours`。
  * 礼物处于可赠送白名单或已验证可安全赠送。
* 本次可赠送总数应来自筛选后的临期候选行，而不是某个礼物的全部库存。
* 日志应区分：
  * 总背包礼物数。
  * 临期候选礼物数。
  * 被跳过的未临期礼物。
  * 被跳过的无过期时间礼物。
  * 被跳过的不可确认可赠送礼物。

### Gift Scope

* 第一版可以默认只自动赠送荧光棒 `id=268`，但数据结构和 UI 文案必须为“可赠送临期礼物”留扩展空间。
* 如果纳入其他礼物，必须先验证：
  * 当前项目现有送礼接口是否支持该 `giftId`。
  * 斗鱼是否按预期扣减临期行/最早过期库存。
  * 失败时是否会影响后续同批任务。
* 非荧光棒礼物建议先做只读展示，再通过配置显式启用自动释放。

### Sending Strategy

* 按临期候选行汇总出本次释放预算。
* 若同一 `giftId` 有多个临期行，可以按 `giftId` 汇总数量，但必须在日志里说明来源行和最早过期时间。
* 如果送礼接口不能指定批次，则实现应明确说明“按临期候选数量赠送，但实际扣减顺序由斗鱼决定”。
* 不应默认执行“清空背包”或“清空该礼物全部库存”。
* 如果接口无法保证扣减临期批次，仍然可以保守地只发送临期数量，避免过度释放。

### WebUI / Observability

* 临期任务页面或概况页应展示背包临期明细：
  * 礼物名称 / ID
  * 数量
  * 过期时间
  * 剩余时间
  * 是否进入当前阈值
  * 是否会被自动释放
  * 跳过原因
* 允许用户通过配置控制：
  * 临期阈值小时数。
  * 自动释放礼物范围：默认只荧光棒，未来可扩展白名单。
  * 是否只展示非荧光棒临期礼物，不自动赠送。

## Acceptance Criteria

* [ ] 背包解析能返回行级礼物明细，而不只返回荧光棒总数和最早过期时间。
* [ ] 临期任务只把进入阈值的可赠送行数量纳入本次赠送预算。
* [ ] 未临期库存不会因为同一礼物存在更早过期行而被全部纳入赠送预算。
* [ ] 无过期时间或不可确认可赠送的礼物不会自动赠送。
* [ ] WebUI 或日志能说明每个候选礼物为什么会赠送或跳过。
* [ ] 当前单行荧光棒样本下，行为与原 MVP 等价：进入阈值后赠送当前可见荧光棒数量。
* [ ] 多行荧光棒样本下，只赠送进入阈值的行数量。
* [ ] Lint / type-check 通过。

## Out of Scope

* 不实现“清空背包”。
* 不默认自动释放所有礼物。
* 不依赖 DouyuEx 的 `batchInfo` 作为稳定批次合同。
* 不承诺能指定斗鱼实际扣减哪一个批次；除非后续实测证明送礼接口支持。
* 不引入浏览器自动化。

## Open Questions

* 非荧光棒礼物第一版是否只读展示，还是允许用户配置白名单后自动释放？
* 如果同一礼物有多个临期行，且送礼接口无法指定批次，是否接受“只发送临期数量，但扣减顺序由斗鱼决定”的策略？
* 是否需要新增“背包明细调试导出”，用于收集真实多行、多过期时间样本？

## Suggested Implementation Notes

* 先在 `src/core/api.ts` 增加结构化背包解析函数，例如 `getBackpackStatus()`，再让 `getGiftStatus()` 复用它生成兼容摘要。
* 临期筛选逻辑应放在 `src/core`，Docker job 只负责任务编排、日志和配置读取。
* UI 不要直接展示原始背包 payload；只展示归一化后的安全字段。
* 失败语义沿用现有约定：背包接口失败不是 `0` 库存，必须展示错误状态。
* 需要新增或更新测试覆盖：
  * 单条荧光棒。
  * 多条荧光棒，只有一条临期。
  * 多条不同礼物。
  * 无绝对过期时间。
  * 非白名单礼物。
