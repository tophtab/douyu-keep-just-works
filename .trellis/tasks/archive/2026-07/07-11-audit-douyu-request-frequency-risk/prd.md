# 审查斗鱼访问频率与风控风险

## Goal

系统梳理项目对斗鱼相关域名和接口的全部访问路径，估算默认配置与可配置极端情况下的请求频率、并发和突发流量，判断是否存在因轮询、重试、任务重叠或用户手动触发而导致账号、Cookie、IP 或设备维度风控的风险，并给出有证据锚点的风险清单与整改建议。

## Background

- 用户关心当前项目访问斗鱼的频率是否安全，以及是否可能因访问过高触发风控。
- 仓库为 TypeScript 单仓库，斗鱼访问逻辑主要位于 `src/core/`，任务调度与手动触发入口主要位于 `src/docker/`。
- 本轮首先进行只读代码与配置审查，不修改业务代码，不主动向斗鱼发起真实请求。
- 用户已确认“当前”仅指仓库代码、默认配置和当前工作区的本地配置，不纳入其他 NAS/服务器实例。
- 跨会话记忆搜索未命中直接相关讨论；OpenCode 历史当前因本地读取器限制不可检索，因此以仓库代码、配置、测试和 Git 历史为准。
- 当前工作区存在 `config/config.json`，但其中 `collectGift`、`keepalive`、`doubleCard`、CookieCloud 均未启用，缺失的 `expiringGift` 与 `yubaCheckIn` 会归一化为未启用；本地 `docker ps` 未发现运行中的容器。因此就此工作区当前状态而言，没有自动任务正在向斗鱼发起请求。
- 实际配置保留了 10 个送礼房间，但对应任务均关闭；这些房间数只影响未来启用后的单次任务请求量，不影响当前静止状态。
- 代码已有同任务互斥、状态缓存、请求合并、送礼间隔和鱼吧随机间隔，但没有全局斗鱼请求限速器，也没有 cron 最小周期约束。

## Requirements

- R1：枚举所有直接或间接访问斗鱼域名/API 的代码路径，记录请求方法、目标、调用者和触发条件。
- R2：梳理所有自动触发来源，包括 cron、启动时执行、周期轮询、Cookie 恢复/同步、保活、签到、礼物、双倍卡及状态同步。
- R3：梳理所有放大访问量的机制，包括循环、批量账号/房间/主播、并发、重试、超时后重入、任务重叠和手动强制刷新。
- R4：基于默认配置计算或给出可验证的请求频率上界；对依赖账号数、房间数或动态分页的路径给出参数化公式。
- R5：检查是否存在限速、随机抖动、指数退避、并发锁、幂等保护、缓存和请求合并，并判断其实际覆盖范围。
- R6：按严重程度列出风控风险，区分账号/Cookie、IP、设备或接口维度，并为每项风险提供文件与行号证据。
- R7：给出优先级明确的缓解建议，区分无需改代码的安全配置建议与需要改代码的防护措施。

## Acceptance Criteria

- [x] 输出斗鱼请求入口清单，覆盖仓库内所有网络请求封装和调用路径。
- [x] 输出当前本地状态、默认配置、典型 10 房间场景和最坏可配置场景的频率/并发分析；无法精确计算的部分明确变量与假设。
- [x] 明确回答当前默认配置是否安全、风险在什么条件下显著上升，以及最值得优先处理的风险点。
- [x] 每个中高风险结论均包含可复核的 `file:line` 证据和触发链路。
- [x] 检查现有测试是否覆盖频率、并发、重试与任务互斥合同，并指出覆盖缺口。
- [x] 审查过程不向斗鱼发起真实网络请求，不改动业务代码。

## Out of Scope

- 不验证或推测斗鱼未公开的精确风控阈值；结论将基于请求行为、常见反自动化信号和保守工程判断。
- 不登录真实账号、不抓取线上或其他机器的容器流量、不进行压力测试或阈值探测。
