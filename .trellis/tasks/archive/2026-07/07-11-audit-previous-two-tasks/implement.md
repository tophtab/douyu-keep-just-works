# Audit Execution Plan

1. 固定证据范围
   - 读取两次归档任务的全部规划、研究、报告和任务元数据。
   - 提取对应 Codex 会话的决策与完成记录。
   - 建立提交到需求的映射表。

2. 审计第一次任务的交付完整性
   - 比较历史重建前后的可用 Git 对象、提交树、归档和 journal 引用。
   - 逐项检查送礼延迟、依赖修复、配置模块重命名、三个旧转换移除、配置不变量保留和凭证恢复阶段化。
   - 检查修改后的现有测试是否仍能约束这些行为，以及有哪些关键分支没有直接覆盖。

3. 审计第二次任务的交付完整性
   - 检查 `did` 复用、日志懒加载、配置保存助手及 CI 门禁的当前数据流。
   - 审阅 5 项新增行为测试的断言质量和失效检测能力。
   - 确认第二次任务没有破坏第一次任务的约束。

4. 运行当前质量门禁
   - `npm run lint`
   - `npm run type-check`
   - `npm run test:contracts`
   - `npm run build:docker`
   - `npm audit --omit=dev --audit-level=moderate`
   - 视审查结果运行聚焦的只读诊断命令。

5. 形成审查报告
   - 按严重程度列出发现，附文件:行、提交和可复现证据。
   - 分别判定两次任务为成功、部分成功或失败。
   - 单列第一次任务的历史重建完整性结论、当前残余风险和建议后续任务。

6. 收尾
   - 确认本任务未修改业务代码。
   - 将审查结论写入任务内 `review.md`，提交用户审阅后再按 Trellis 流程归档。
